# 01 — Repository Structure Map

This map describes the current top-level layout of the VoxeraOS repository.
It is generated by inspection of the working tree (see
`directory_tree.txt` for the full recursive listing).


Subpackages:

- **`voxera.brain/`** — LLM client backends: `base`, `fallback`, `gemini`, `openai_compat`, `json_recovery`.
- **`voxera.core/`** — Queue/object model, daemon, mission planner, intent classifiers, capability/execution evaluators, hygiene/recovery/reconcile, queue contracts, artifact helpers. See file 03 and 04.
- **`voxera.panel/`** — FastAPI panel app and one route module per surface; templates in `panel/templates/`. See file 02.
- **`voxera.vera/`** — Vera control layer: session store, drafting, handoff, evidence review, investigation, weather, brave_search, prompt, reference resolver, context lifecycle, linked completions. See file 05.
- **`voxera.vera_web/`** — Vera FastAPI chat web app and supporting modules: `app.py`, `chat_early_exit_dispatch`, `conversational_checklist`, `draft_content_binding`, `execution_mode`, `preview_content_binding`, `response_shaping`. Templates in `vera_web/templates/index.html`.
- **`voxera.skills/`** — Skill registry, runner, execution helpers, arg normalizer, path boundaries, result contract.
- **`voxera.audio/`** — Stub audio module (`README.md`, `__init__.py`).
- **`voxera.voice/`** — `flags.py`, `input.py`, `output.py`, `models.py` (voice foundation seam).
- **`voxera.data/`** — `openrouter_catalog.json` cached data.

## `src/voxera_builtin_skills/`

Each module is a single skill entrypoint exposing a `run(...)` function that
matches its corresponding `manifest.yml`. Categories observed in the current
tree:

- Files: `files_copy`, `files_copy_file`, `files_delete_file`, `files_exists`,
  `files_find`, `files_grep_text`, `files_list_dir`, `files_list_tree`,
  `files_mkdir`, `files_move`, `files_move_file`, `files_read_text`,
  `files_rename`, `files_stat`, `files_write_text`.
- Clipboard: `clipboard_copy`, `clipboard_paste`.
- System / host: `system_status`, `host_info`, `process_list`, `service_status`,
  `recent_service_logs`, `disk_usage`, `memory_usage`, `set_volume`,
  `terminal_run_once`, `window_list`, `load_snapshot`.
- App / URL: `open_app`, `open_url`.
- Sandbox: `sandbox_exec`.

The `skill_manifest_inventory.json` records the manifest side of the same set.

## `skills/` — manifest tree

Each skill has its own directory containing `manifest.yml`. Top-level
groups: `clipboard/`, `files/`, `sandbox/`, `system/`. The tree currently
holds 31 manifests; see `skill_manifest_inventory.json` for full metadata
(id, capabilities, risk, exec_mode, fs_scope, args, output_artifacts,
output_schema).

## `missions/` — mission JSON definitions

- `sandbox_smoke.json` — basic sandbox exec, network disabled.
- `sandbox_net.json` — sandbox exec with `network=true` to exercise the approval flow.
- `README.md` — operator notes.

Mission templates also live in code at `src/voxera/core/missions.py`
(see file 04).

## `tests/`

Top-level test files exclusively (no nested package). Notable groups:

- Queue / daemon: `test_queue_daemon*.py`, `test_queue_constitution_contracts.py`,
  `test_queue_execution_contracts.py`, `test_queue_artifact_minimum_regression.py`,
  `test_queue_job_intent.py`, `test_queue_result_consumers.py`.
- CLI surfaces: `test_cli_*` (queue, contract snapshot, version, run args, secrets, config snapshot, …).
- Vera: `test_vera_*` (chat reliability, contextual flows, draft revision, hidden compiler, investigation derivations, live path characterization, panel, preview materialization, preview submission, runtime validation, session characterization, web).
- Skills: `test_files_*`, `test_builtin_skills_terminal_run_once.py`, `test_sandbox_exec_*`, `test_skill_metadata_and_runners.py`, `test_skill_result_payloads.py`, `test_runner.py`, `test_registry.py`, `test_arg_normalizer.py`.
- Operator/panel: `test_panel*.py`, `test_operator_assistant_queue.py`, `test_operator_contract_guardrails.py`.
- Doctor / health / docs / config: `test_doctor.py`, `test_health.py`, `test_health_snapshot_isolation.py`, `test_docs_consistency.py`, `test_config_*`.
- Golden / red-team / mypy ratchet: `test_golden_surfaces.py`, `test_security_redteam.py`, `test_mypy_ratchet.py`.
- Helper module: `vera_session_helpers.py`.

Snapshot baselines live in `tests/golden/`.

## `scripts/`

## `tools/`

- `golden_surfaces.py` — golden snapshot driver used by `make golden-update/golden-check`.
- `mypy-baseline.txt` — current mypy ratchet baseline.

## `deploy/systemd/user/`

Three user-mode systemd unit templates installed by `make services-install`:

- `voxera-daemon.service` — runs `voxera daemon` (queue + missions).
- `voxera-panel.service` — runs `voxera panel --host 127.0.0.1 --port 8844`.
- `voxera-vera.service` — runs `uvicorn voxera.vera_web.app:app --host 127.0.0.1 --port 8790`.

`@VOXERA_PROJECT_DIR@` is templated to the absolute repo path during install.

## `systemd/`

Older standalone units retained for reference: `voxera-core.service`,
`voxera-panel.service`. The active install path is `deploy/systemd/user/`.

## `config-templates/`

Sample runtime configuration files for new operators (no secrets).

## `docs/`

Long-form architecture, queue model, security, ops, and roadmap documents
plus this regenerated bundle. See file 10 for the full informational map.
